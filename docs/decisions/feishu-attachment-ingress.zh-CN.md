# 飞书附件入口与本地暂存边界

英文原文：`docs/decisions/feishu-attachment-ingress.md`

另见：

- `docs/architecture/feishu-codex-design.zh-CN.md`：当前架构与仓库边界
- `docs/contracts/codex-permissions-model.zh-CN.md`：`sandbox`、`approval`、writable roots 的语义
- `docs/contracts/group-chat-contract.zh-CN.md`：群激活、群聊工作态与触发边界
- `docs/decisions/feishu-output-images.zh-CN.md`：反方向能力，即 Codex 图片结果回到飞书的边界

## 1. 问题陈述

用户希望在飞书里直接发送附件，然后继续通过 `feishu-codex` 与本地 Codex 协作。

本仓库当前的现状是：

- 文件消息仍被显式拒绝
- app-server turn 输入仍以纯文本为主
- 复杂文件解析能力并不应该由 `feishu-codex` 自己长期维护

如果把 PDF、Office、OCR、音视频转写、压缩包处理都内建进本仓库，会引入：

- 不清晰的产品边界
- 高维护成本
- 大量与本地环境、MCP、上游模型能力耦合的歧义

因此，这里需要先定义一条更窄、更稳定的设计边界。

## 2. 决策摘要

本仓库对飞书附件能力的第一阶段设计决策是：

1. `feishu-codex` 只负责**接收可下载的飞书附件消息、下载资源、暂存到当前工作目录、并把本地路径交给 Codex**。
2. 复杂文件解析不是本项目职责；应交给：
   - Codex 原生能力
   - 用户配置的 MCP / Apps
   - 本地环境里已经可用的解析工具链
3. 图片是唯一需要额外“升级”的附件类型：
   - 除保存到本地外，还应在 turn 输入中提升为 `localImage`
4. 非图片附件统一按“本地文件路径”处理：
   - `feishu-codex` 不承诺模型一定能直接理解该文件
   - 只承诺文件已被保存到本地，且路径会被明确交给 Codex
5. 本仓库**不自动**做以下行为：
   - OCR
   - 文本抽取
   - 文件格式转换
   - 自动解压
   - 自动执行附件
6. 飞书 API 本身无法稳定支持的附件类型，必须显式拒绝，不做模糊降级。

## 3. 这条边界为什么更合适

这条边界的核心优点是把责任收紧到清晰的两层：

- `feishu-codex`
  - 负责消息入口、权限/群激活校验、下载、暂存、turn 绑定
- Codex / MCP / 本地环境
  - 负责“这个文件怎么理解、怎么转换、怎么分析”

这样做的好处是：

- 架构更清晰：附件入口和文件解析不再混在一起
- 维护成本更低：本仓库不需要自己实现文档解析生态
- 行为更明确：支持的是“把文件带进本地工作区”，不是“保证所有格式都能直接理解”
- 更符合 fail-closed：不能稳定支持的类型直接拒绝

## 4. 支持范围

本设计里的“附件支持”只指**飞书消息资源可下载**的附件类型。

### 4.1 第一阶段可接入的消息类型

| 飞书消息类型 | 是否下载 | 进入 Codex 的方式 | 本仓库承诺 |
| --- | --- | --- | --- |
| `image` | 是 | 本地文件 + `localImage` | 保存成功，并作为图片输入交给 Codex |
| `file` | 是 | 本地文件路径 | 保存成功，并把路径交给 Codex |
| `audio` | 是 | 本地文件路径 | 保存成功；是否可理解交给 Codex / MCP / 本地工具 |
| `media` | 是 | 本地文件路径 | 保存成功；是否可理解交给 Codex / MCP / 本地工具 |

### 4.2 明确不在第一阶段合同内的类型

| 类型 | 不纳入原因 |
| --- | --- |
| `folder` | 飞书 API 只能拿到 key 和名称，不能基于 API 下载 |
| `sticker` | 飞书消息资源接口不支持下载 |
| `merge_forward` 内部子附件 | 飞书消息资源接口不支持下载合并转发子消息资源 |
| `interactive` 卡片内资源 | 飞书消息资源接口不支持下载卡片资源 |

### 4.3 其他说明

- `text`、`post`、`interactive` 的文本提取仍属于现有文本消息路径，不属于本设计里的“附件入口”。
- `file` 类型里可以包含 PDF、DOCX、XLSX、PPTX、ZIP、代码文件、CSV 等。
- 对这些格式，本仓库的承诺只是“保存到本地并把路径交给 Codex”，**不承诺**解析一定成功。

## 5. 生命周期合同

### 5.1 附件消息本身不直接启动 turn

当用户发送一条附件消息时：

- 先完成权限/群聊触发校验
- 下载附件到当前工作目录下的固定附件目录
- 记录一条 pending attachment 状态
- 回复用户“附件已保存，继续发送文字说明即可”

附件消息本身不应直接启动 Codex turn。

原因：

- 大多数附件仍需要用户补充意图
- 避免“只发一个文件就突然执行”的误触发
- 让多个附件后接一条文本说明成为自然路径

### 5.2 消费规则

后续由**同一发送者、同一 chat、同一 thread 语义**下的下一条文本消息消费当前 pending 附件集合。

更具体地说：

- `p2p`：按 `sender + chat`
- 群聊主聊天流：按 `sender + chat`
- 群聊话题：按 `sender + chat + thread_id`

同一 key 下、尚未消费的 pending 附件应按接收顺序整体消费。

如果该 pending 集合里任一附件本地已缺失，或者已经不再属于当前工作目录，
则必须整组 fail-closed，而不是只拿剩余那一部分继续执行。

### 5.3 群聊隔离规则

在群聊中：

- A 发送的附件绝不能被 B 的文本消费
- pending 附件不能只按 chat 级别共享
- 群聊附件入口仍要服从现有工作态、群激活与触发规则

### 5.4 群聊历史回捞与附件入口的关系

群聊历史回捞与附件入口虽然都运行在群聊工作态、群激活、话题 scope
这些外层规则之下，但它们不是同一条语义路径。

这里的正式边界是：

- 群聊历史回捞只负责恢复**文本讨论上下文**
- 附件是否已下载到本地、是否仍可用、是否已被后续文本消费，只由附件入口的 pending attachment 状态决定
- 历史回捞不负责“恢复附件可用性”
- 历史回捞也不负责“重建附件消费状态”

因此：

- 即使群上下文中出现了文件名或文件类占位文本，也不应被理解为对应附件当前仍然可用
- 如果后续 rollout 启用附件入口，群聊 `assistant` 上下文日志与历史回捞都不应承担附件生命周期状态
- 附件生命周期应保持为独立状态层，而不是作为群文本上下文的一部分被隐式编码

### 5.5 过期规则

pending 附件必须有 TTL。

到期后：

- 不再自动绑定到后续文本
- 应允许清理本地状态
- 用户再次提及时，应提示“附件已过期，请重新发送”

## 6. 本地暂存合同

### 6.1 保存位置

所有可下载附件统一保存到当前工作目录下的固定子目录：

- `_feishu_attachments/`

不直接散落到工作目录根下。

这样做是为了：

- 降低对仓库根目录的污染
- 让用户更容易理解这些文件来自飞书入口
- 使清理与排障有稳定路径

### 6.2 文件命名

暂存文件名不能只依赖用户上传时的原始文件名。

实现上应至少保证：

- 保留原始扩展名
- 文件名做路径安全清洗
- 路径具备抗冲突标识，例如时间戳、`message_id`、hash 等

目标是避免：

- 覆盖用户现有文件
- 不同消息同名文件互相踩写
- 恶意文件名破坏目录结构

### 6.3 与权限模型的关系

该设计有一个刻意的产品语义：

- 一旦附件被保存到当前工作目录，就把它视为**当前工作区的一部分**

因此：

- `read-only` 下，Codex 可以读取这些文件，但不能直接修改
- `workspace-write` 下，这些文件会像工作区内普通文件一样可被编辑
- `danger-full-access` 下，则服从更宽的宿主权限边界

这不是副作用，而是本设计为了换取更简单体验而接受的边界。

它还有一个直接推论：

- 如果 pending 附件尚未消费前工作目录已经变化，这批 pending 附件必须失效，
  不能再被重绑定到新的工作区中
- 这里的“失效”只指失去自动消费资格，不等于删除原工作区里的本地文件
- 已下载文件仍保留在原工作区，供人工排查、手工复用或显式路径引用

## 7. Codex 输入合同

### 7.1 图片

图片附件在启动 turn 时应做两件事：

1. 作为普通本地文件保留在 `_feishu_attachments/`
2. 在 `turn/start` 输入中提升为 `localImage`

这样既保留本地路径，又能利用 Codex 对图片的原生输入支持。

### 7.2 非图片附件

非图片附件不做额外模态升级。

启动 turn 时：

- 通过文本把附件的本地绝对路径、显示名、类型说明交给 Codex
- 由 Codex 决定是否直接读取、是否调用本地工具、是否转交 MCP / Apps

这里的关键点是：

- `feishu-codex` 不负责判断“这个 PDF / DOCX / MP4 到底该怎么读”
- `feishu-codex` 只负责把文件可靠地带到本地工作区，并把路径明确交给 Codex

## 8. 显式不做的事

第一阶段不应默认实现下列行为：

- 自动 OCR
- 自动 PDF / Office 文本抽取
- 自动音频转写
- 自动视频抽帧
- 自动解压压缩包
- 自动执行脚本或二进制附件
- 自动根据扩展名猜测最佳解析路径

如果管理员或用户需要这些能力，应通过：

- MCP / Apps
- 本地已安装工具
- 明确提示 Codex 调用某个环境命令

来完成。

## 9. 错误与拒绝语义

不支持或失败时，必须优先给出**原因明确**的文案，而不是泛化成“文件处理失败”。

至少要区分：

- 类型不支持：
  - 文件夹
  - 表情包
  - 合并转发子附件
  - 卡片资源
- 飞书资源下载限制：
  - 超过 100 MB
  - 资源对当前机器人不可见
  - 外部群 / 保密 / 防泄密模式限制
- 本地暂存失败：
  - 目录不可写
  - 磁盘空间不足
  - 文件名冲突未能安全重命名

## 10. 管理员与用户职责

### 10.1 管理员职责

- 选择合适的工作目录；因为附件会进入当前 workspace
- 根据需要配置 MCP / Apps / 本地工具链
- 根据权限预设决定附件是否允许被原地编辑
- 配置或接受附件 TTL、清理策略、磁盘占用上限

### 10.2 用户职责

- 发送附件后，再发送一条明确文字说明
- 对复杂文件格式，如果需要更强解析能力，应配合环境或 MCP 使用
- 不应把“文件成功保存到本地”理解成“模型保证能直接理解该格式”

## 11. 实现清单

后续真正实现时，至少应覆盖以下项目：

1. 把当前 file-only 入口扩成 attachment ingress，纳入：
   - `image`
   - `file`
   - `audio`
   - `media`
2. 抽出统一的消息资源下载能力，能区分：
   - `type=image`
   - `type=file`
3. 把附件统一保存到 `cwd/_feishu_attachments/`
4. 引入 pending attachment 状态，并按 `sender + chat + thread` 隔离
5. 附件消息只下载与确认，不直接启动 turn
6. 后续文本消息消费 pending 附件集合
7. 扩宽 adapter / turn 输入，使图片能够进入 `localImage`
8. 非图片附件通过明确的本地路径说明注入 turn 文本
9. 对不支持类型与飞书下载失败返回清晰错误
10. 增加 TTL / 清理机制
11. 补回归测试，至少覆盖：
    - 单聊图片
    - 单聊普通文件
    - 群聊图片/文件
    - 群聊跨用户隔离
    - 不支持类型拒绝
    - TTL 过期

## 12. 相关实现入口

当前实现主要落在：

- `bot/file_message_domain.py`
- `bot/feishu_bot.py`
- `bot/adapters/base.py`
- `bot/adapters/codex_app_server.py`
- `bot/prompt_turn_entry_controller.py`
- `bot/stores/pending_attachment_store.py`

当前仓库已经按本文边界落地第一阶段附件入口：

- 支持 `image`、`file`、`audio`、`media`
- 附件先下载到 `cwd/_feishu_attachments/`
- 后续同一 `sender + chat + thread` 语义下的文本消息消费 pending 附件
- 图片进入 `localImage`，其他附件通过本地路径说明注入 turn 文本
- 支持 TTL 过期清理与回归测试覆盖
