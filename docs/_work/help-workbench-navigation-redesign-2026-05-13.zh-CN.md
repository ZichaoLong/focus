# 飞书帮助工作台重构草案（2026-05-13）

本文记录 `/help` / `/h` 导航重构的已对齐方案与实现计划。

这是一份工作草案，不直接替代当前正式合同：

- 正式合同仍以 `docs/contracts/feishu-help-navigation*.md` 为准
- 在代码落地并完成测试前，不先把“目标设计”写回正式合同

## 1. 重构目标

当前帮助导航的主要问题不是功能缺失，而是信息架构过度暴露内部抽象：

- 首页按 `当前会话 / 群聊 / 线程 / 运行时 / 身份` 分组
- 用户在实际使用时通常先想到“我要做什么”，而不是“这件事属于哪个抽象层”
- `线程`、`当前会话`、`运行时` 之间容易点错
- 手机飞书里长卡片与三列按钮会进一步放大误触与阅读负担

本次重构目标：

1. 首页先给出精简状态摘要，再给固定工作区
2. 一级导航按正常工作流分组，而不是按内部对象分组
3. 一级工作区页尽量短、小、稳定，便于手机里随时回到首页
4. 高级恢复动作保留在上下文化结果卡中，不强行塞入主工作流

## 2. 已对齐结论

### 2.1 首页结构

`/help` 首页改为“工作台”而不是“主题目录”：

- 标题：`Codex 工作台`
- 内容分两部分：
  - 状态摘要
  - 固定工作区

首页不再引入：

- 动态建议操作页
- `我想...` 输入路由

### 2.2 首页状态摘要

首页状态摘要应尽量压缩到 2 到 3 行，避免手机端展开后占太多空间。

建议字段：

- 当前目录
- 当前线程
- 推送状态：`attached` / `detached`
- 本轮设置摘要：`permissions / model / effort / collab-mode`
- 群聊上下文中可额外显示群工作态

### 2.3 固定工作区

首页固定暴露六个工作区：

- `开始`
- `线程设置`
- `本轮设置`
- `连接状态`
- `群聊设置`
- `更多`

这些名字面向动作，不面向内部抽象。

### 2.4 返回按钮规则

- 一级工作区页：必须有 `返回首页`
- 下级页：只保留 `返回上一页`
- 所有返回按钮都独占一整行

原因：

- 首页是手机端的“收口态”
- 下级页若再放 `返回首页`，会稀释“当前在这条工作流里向上返回”的路径
- 返回按钮独占一行，避免与操作按钮混排造成误触

### 2.5 排版规则

首页与各级导航页不强制三列。

允许的策略：

- 首页：控制为两行六入口；实现时优先用稳定的双列/两行布局
- 一级工作区页：优先双列
- 如果某页按钮文字较长、双列仍拥挤，可退化为单列
- 只有高密度列表或明确需要压缩纵向空间的卡片继续使用三列

典型保留三列的场景：

- `/threads` 线程列表卡

### 2.6 布局能力核实结果

当前代码里已经存在稳定的 `bisected` 双列布局使用，不需要为两列按钮额外造机制。

现有证据：

- `bot/cards.py` 已有 `layout: "bisected"` 用法
- `bot/codex_help_domain.py` 已有 `layout="bisected"` 用法

因此实现层的策略应是：

- 导航首页与一级工作区页优先用 `bisected`
- 若某一页在真实手机端显示仍不理想，再局部退回单列
- 不为了形式统一而强行三列

## 3. 目标 IA 草图

### 3.1 首页：Codex 工作台

首页只包含：

- 精简状态摘要
- 六个固定工作区入口

首页不承载：

- 动态推荐
- 意图搜索
- 低频诊断表单

### 3.2 工作区：开始

解决的问题：

- 开新线程
- 恢复旧线程
- 浏览线程
- 切换目录

一级按钮：

- `新建线程` -> `/new`
- `恢复线程` -> 恢复表单页
- `浏览线程` -> `/threads`
- `切换目录` -> 目录表单页

底部：

- `返回首页`

下级页：

- `恢复线程`
  - 表单提交：`/resume <thread_id|thread_name>`
  - 底部：`返回上一页`
- `切换目录`
  - 表单提交：`/cd <path>`
  - 正文里说明无参数 `/cd` 等价查看当前目录
  - 底部：`返回上一页`

### 3.3 工作区：线程设置

解决的问题：

- 改当前绑定 thread 本身

一级按钮：

- `改 Profile` -> `/profile`
- `改 Memory` -> `/memory`
- `压缩上下文` -> `/compact`
- `重命名` -> 重命名表单页
- `归档当前` -> `/archive`
- `按目标归档` -> 归档表单页

底部：

- `返回首页`

下级页：

- `重命名`
  - 表单提交：`/rename <title>`
  - 底部：`返回上一页`
- `按目标归档`
  - 表单提交：`/archive [thread_id|thread_name]`
  - 底部：`返回上一页`

### 3.4 工作区：本轮设置

解决的问题：

- 改当前飞书会话后续 turn 的行为

一级按钮：

- `权限预设` -> `/permissions`
- `模型` -> `/model`
- `推理强度` -> `/effort`
- `审批策略` -> `/approval`
- `沙箱策略` -> `/sandbox`
- `协作模式` -> `/collab-mode`

底部：

- `返回首页`

说明：

- `approval` / `sandbox` 不再藏到二级页
- `model` / `effort` 虽然可共享一张结果卡，但帮助入口仍分成两个按钮

### 3.5 工作区：连接状态

解决的问题：

- 看当前状态
- 发送前预检
- 控制是否继续接收飞书推送
- 处理常见附着动作

一级按钮：

- `当前状态` -> `/status`
- `发送前检查` -> `/preflight`
- `暂停推送` 或 `恢复当前会话`
  - 当前 attached 时显示 `暂停推送` -> `/detach`
  - 当前 detached 时显示 `恢复当前会话` -> `/attach`
- `附着当前实例` -> `/attach service`
- `更多附着方式` -> 附着方式页

底部：

- `返回首页`

下级页：

- `更多附着方式`
  - `附着当前线程` -> `/attach thread`
  - `附着当前会话` -> `/attach`
  - 底部：`返回上一页`

说明：

- 常见用户动作是“附着整个实例”，因此 `附着当前实例` 直接放一级页
- thread / binding 级附着放在 `更多附着方式`

### 3.6 工作区：群聊设置

解决的问题：

- 看本群是否已启用
- 启用 / 停用本群
- 改群工作态

一级按钮：

- `群聊启用状态` -> `/group`
- `启用本群` -> `/group activate`
- `停用本群` -> `/group deactivate`
- `群工作模式` -> `/group-mode`

底部：

- `返回首页`

说明：

- 这里不再使用“群状态”这种含义不够明确的标签

### 3.7 工作区：更多

解决的问题：

- 身份
- 命令索引
- 低频高级动作

一级按钮：

- `身份信息` -> `/whoami`
- `机器人状态` -> `/bot-status`
- `初始化` -> init 表单页
- `命令索引` -> `/commands`
- `高级操作` -> 高级操作页

底部：

- `返回首页`

下级页：

- `初始化`
  - 表单提交：`/init <token>`
  - 底部：`返回上一页`
- `高级操作`
  - `重置 backend` -> `/reset-backend`
  - `联系人排障` -> debug-contact 表单页
  - 底部：`返回上一页`
- `联系人排障`
  - 表单提交：`/debug-contact <open_id>`
  - 底部：`返回上一页`

## 4. 仍保留的上下文化行为

工作台不会替代已有的上下文化结果卡。

尤其保留：

- `/profile` 结果卡里的 reset / attach 后续动作
- `/memory` 结果卡里的 reset / attach 后续动作
- `/reset-backend` 结果卡里的 attach 后续动作

原因：

- 这些后续动作与刚刚发生的上下文强相关
- 用户完成 thread 设置或 backend reset 后，最自然的下一步仍然是点结果卡
- 主工作台只负责提供常驻入口，不负责抹平所有上下文差异

## 5. 不进入固定工作区首页入口的命令

下列命令仍然不需要出现在固定工作区首页：

- `/help`
- `/h`
- `/pwd`
- `/cancel`

处理方式：

- `/help` / `/h`
  - 它们本身就是入口
- `/pwd`
  - 并入 `切换目录`
- `/cancel`
  - 继续以执行卡按钮为主入口

## 6. 分阶段实现计划

### 阶段 1：帮助导航结构改造

范围：

- 改写 `bot/codex_help_domain.py` 的页面信息架构
- 首页改为“状态摘要 + 六个固定工作区”
- 新增各工作区与其下级页
- 返回按钮规则改为：
  - 一级页 `返回首页`
  - 下级页 `返回上一页`

验证：

- 单元测试覆盖各页可达性
- 表单回落逻辑仍正常

### 阶段 2：帮助页按钮排版收敛

范围：

- 首页与一级工作区页优先使用双列
- 按实际按钮文字长度，必要时局部改单列
- 保持高密度卡片（如 `/threads`）继续走三列路径

验证：

- 单测保证布局字段符合预期
- 手工在手机飞书里核对行数与按钮截断情况

### 阶段 3：同步正式合同与命令矩阵

代码行为稳定后，再同步更新：

- `docs/contracts/feishu-help-navigation*.md`
- `docs/contracts/feishu-command-matrix*.md`

同步原则：

- 先改代码与测试
- 再让正式合同反映新事实
- 这份 `_work` 草案保留为设计与 rollout 记录

## 7. 本次实现前需要保持的兼容策略

在真正改代码时，建议保留一段时间的旧 topic alias 兼容。

例如：

- 旧：`/help chat`
- 新：可内部跳到 `连接状态` 或 `开始`

是否保留以及保留多久，可以在实现阶段再定，但不建议在第一版重构里直接硬删旧 alias。
